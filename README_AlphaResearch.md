# Alpha Research Platform

> Manager-ready quantitative research platform — alpha signal generation, portfolio construction, backtesting, real-time execution engine, and PyTorch predictive modeling with C++ execution simulation.

---

## Overview

This platform implements the full quant research pipeline from raw signal to simulated execution:

```
Raw Data → Signal Generation → Alpha Combination → Portfolio Construction
    → Risk Management → Backtesting → Real-Time Engine → C++ Execution Sim
```

It is designed to mirror the workflow used by systematic trading desks: signals are developed and validated in research mode, then promoted to the live execution engine with minimal code changes.

---

## Components

### 1. Signal Generation (`signals/`)

Implements a library of alpha signals across multiple categories:

| Category | Signals |
|---|---|
| **Momentum** | Cross-sectional momentum, time-series momentum, idiosyncratic momentum |
| **Mean Reversion** | Pairs z-score, Ornstein-Uhlenbeck residuals |
| **Microstructure** | Order imbalance, bid-ask spread dynamics, trade flow toxicity |
| **ML-Based** | PyTorch LSTM return predictor, gradient-boosted rank signal |

Each signal is normalised to z-scores cross-sectionally per timestep.

### 2. Alpha Combination

Signals are combined using an IC-weighted ensemble (Information Coefficient weighting):

```python
def ic_weighted_composite(signals: pd.DataFrame, 
                           returns: pd.Series, 
                           window: int = 60) -> pd.Series:
    ics = {col: signals[col].rolling(window).corr(returns) 
           for col in signals.columns}
    ic_df = pd.DataFrame(ics).fillna(0)
    weights = ic_df.div(ic_df.abs().sum(axis=1), axis=0)
    return (signals * weights).sum(axis=1)
```

This dynamically down-weights signals that have lost predictive power.

### 3. Portfolio Construction (`portfolio/`)

Mean-variance optimisation with transaction cost awareness:

- **Objective:** Maximise `α·w - λ·w'Σw - c·|Δw|`
- **Constraints:** Gross exposure, sector neutrality, max position size
- **Solver:** `cvxpy` with OSQP backend

Long-short equity portfolio targeting market-neutral exposure.

### 4. Backtesting Engine (`backtest/`)

Event-driven backtester with realistic market impact modelling:

```python
class Backtest:
    def run(self, signals, prices, universe):
        for date in self.trading_days:
            alpha   = self.alpha_model.score(signals.loc[date])
            weights = self.optimizer.solve(alpha, self.cov_model.predict(date))
            trades  = weights - self.current_weights
            fills   = self.execution_model.fill(trades, prices.loc[date])
            self.portfolio.update(fills)
            self.record(date)
```

Accounts for: bid-ask spread, market impact (square-root model), borrow costs, and realistic slippage.

### 5. PyTorch Predictive Model (`models/`)

LSTM trained on lagged returns and microstructure features to predict 5-day forward returns:

```
Input:  [returns_t-20:t, volume_t-20:t, spread_t-20:t, imbalance_t-20:t]
Layer:  LSTM(hidden=128, layers=2, dropout=0.2)
Layer:  Linear(128 → 64) + ReLU
Output: Linear(64 → 1)  [predicted return rank]
```

Trained with Huber loss on cross-sectional ranks (rank-IC optimisation). Walk-forward validation with no lookahead.

### 6. Real-Time Engine (`realtime/`)

Asyncio-based engine that simulates live signal computation and order routing:

```
Market data → Feature computation → Signal scoring → Risk check → Order generation
     ↑                                                                    ↓
     └──────────────── Position update ←──── Simulated fill ←────────────┘
```

Designed to be latency-aware: signals are computed incrementally (no full recompute per tick).

### 7. C++ Execution Simulation

Connects to the [Low-Latency Trading Engine](https://github.com/nisgemML/Low-Latency-Trading-Engine-) via a shared message format. Orders generated by the Python research engine are serialised and fed to the C++ matching engine for realistic fill simulation, including queue position and partial fill modelling.

## Design Philosophy

The codebase is intentionally structured to match how systematic trading desks are organised:

- **Research ↔ Production parity:** The same signal classes run in both backtest and live modes. No reimplementation.
- **Testability:** Every component has a `score(data)` interface that returns a pd.Series of ranks or z-scores.
- **Transparency:** Every signal has an `explain()` method that decomposes its contribution.
